## Philosophy

> Privacy first. Self-hosted first. No paywalls. No lock-in. Forever free.

Before contributing, please internalise these principles:

- **Zero-friction UX is sacred.** The default path is `PROVIDER=auto` → prompt → done. Do not add mandatory configuration steps to the default experience.
- **Build in dependency order.** Foundation contracts (schemas, typed errors, tool interfaces) before runtime features; runtime before UI polish.
- **Self-hosted is the primary deployment target.** Cloud-only features are secondary.
- **Any checked item in the roadmap must map to real code.** No aspirational checkmarks.

Read [docs/STRATEGY_AND_GUARDRAILS.md](docs/STRATEGY_AND_GUARDRAILS.md) before proposing architectural changes.

## How to Get Involved

### Good first issues

Look for issues labelled `good first issue` — these are well-scoped, self-contained tasks with clear acceptance criteria. A failing test, a missing type annotation, a docs fix, or a new tool are all great starting points.

### Roadmap items

The definitive feature list lives in [docs/ROADMAP_FEATURES_V2.md](docs/ROADMAP_FEATURES_V2.md). Items marked `[ ]` and not labelled `[-]` (deferred) are open for contribution. Check that no one else has an open PR for the item before starting.

### Bug reports

If you find a bug, open an issue using the [bug report template](.github/ISSUE_TEMPLATE/bug_report.md). Include reproduction steps, environment details, and the exact error output.

### Feature requests

Use the [feature request template](.github/ISSUE_TEMPLATE/feature_request.md). Proposals that fit the roadmap and align with the philosophy are prioritised.

## Development Setup

### Prerequisites

- Python 3.10+
- Docker + Docker Compose (for full-stack local testing)
- An API key for at least one provider (or run Ollama locally — no key needed)

### Local setup

```bash
git clone https://github.com/The-No-Hands-company/nexus-ai.git
cd nexus-ai

# Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Copy and configure environment
cp .env.example .env
# Edit .env — add at least GROQ_API_KEY or GEMINI_API_KEY

# Run the development server
uvicorn main:app --reload --port 8000
```

Open [http://localhost:8000](http://localhost:8000).

### Docker Compose (full stack)

```bash
cp .env.example .env
docker compose up -d
```

This starts Nexus AI and an Ollama sidecar. Pull a model after startup:

```bash
docker exec ollama ollama pull qwen2.5:7b
```

## Making Changes

1. **Fork** the repository and create a branch from `main`:
   ```bash
   git checkout -b feat/my-feature
   # or
   git checkout -b fix/issue-123
   ```

2. **Make your changes.** Keep each commit focused on one concern.

3. **Test your changes** manually and, where applicable, with automated tests:
   ```bash
   pytest tests/
   ```

4. **Lint and type-check:**
   ```bash
   ruff check .
   mypy .
   ```

5. **Open a pull request** against `main`.

## Pull Request Guidelines

- **One PR per logical change.** Don't bundle unrelated fixes.
- **Fill in the PR template.** Link the related issue, describe what changed and why, and add testing notes.
- **Keep diffs readable.** Avoid reformatting files unrelated to your change.
- **No paywall gates.** Features must be accessible without a paid plan.
- **No opt-out telemetry.** Do not add any tracking that isn't explicitly opt-in.
- **Backwards compatible by default.** Breaking changes require a discussion issue first.
- **All new public endpoints need contract tests.** See [docs/ROADMAP_FEATURES_V2.md § Sprint A](docs/ROADMAP_FEATURES_V2.md) for the testing pattern.

### PR size guidelines

| Size | Guideline |
|---|---|
| XS (< 50 lines) | Ship immediately after review |
| S (50–200 lines) | Target 1–2 review cycles |
| M (200–500 lines) | Split if possible; requires architecture comment |
| L (500+ lines) | Needs a design issue first |

## Issue Guidelines

- **Search before opening.** Your issue may already exist.
- **One issue per problem.** Don't bundle multiple bugs.
- **Include the environment.** Python version, OS, Docker or bare-metal, provider in use.
- **Attach logs.** Paste the relevant stack trace or error output.

### Labels

| Label | Meaning |
|---|---|
| `bug` | Something is broken |
| `enhancement` | New feature aligned with the roadmap |
| `good first issue` | Suitable for new contributors |
| `docs` | Documentation only |
| `security` | Security-sensitive — handled via [SECURITY.md](SECURITY.md) first |
| `breaking` | Introduces a breaking change |
| `deferred` | Acknowledged but not planned for current phase |

## Commit Convention

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

**Types:** `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`

**Examples:**

```
feat(router): add DeepSeek provider with reasoning_content normalisation
fix(tools): clamp run_command CPU timeout to 60s
docs(contributing): add PR size guidelines
refactor(agent): extract tool dispatch into separate module
```

## Style Guide

### Python

- **Formatter:** `ruff format` (Black-compatible, line length 100)
- **Linter:** `ruff check`
- **Type checker:** `mypy` — all new code must pass with no new errors
- **Import order:** stdlib → third-party → local (ruff enforces this)
- **No unused imports.** ruff will catch them.
- **Docstrings:** Google-style for public functions. Not required for private helpers.

### General

- Prefer explicit over implicit.
- Error messages must be actionable — tell the user what to do, not just what went wrong.
- Do not add `print()` statements to production code; use the `logging` module.
- New environment variables must be documented in `.env.example` and in the README env table.
